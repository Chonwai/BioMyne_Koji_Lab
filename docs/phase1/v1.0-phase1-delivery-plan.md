## 1. Objective

本版本計劃的目標，是把 BioMyne Koji Phase 1 的規劃文件，轉化成一個可交給工程 owner 啟動的實作 repository baseline。這個版本不是交付完整產品，而是交付一個能支持 **Phase 1 intelligence loop 驗證** 的專案骨架、資料層、配置層、運維層與觀察層。

## 2. Scope

### In Scope

- 在 `BiomyneKoji/biomyne-koji` 內建立 Phase 1 repository structure
- 建立 SQL migrations baseline
- 建立 `.env.example`
- 建立 biotech source manifest template
- 建立 Hermes profile / skill scaffold
- 建立 Langfuse docker compose baseline
- 建立 operations runbook
- 建立 bootstrapping / validation scripts skeleton
- 建立 repo-level README 與 Phase 1 execution guide

### Out of Scope

- 真正的 FastAPI BFF
- Next.js dashboard
- GraphDB / Neo4j
- Crawl4AI deployment
- production-grade auth/RBAC
- public API surface
- multi-tenant design
- full frontend UI

## 3. Business Problem

BioMyne 已完成 Koji 的策略、架構與工程規格文件，但尚未把這些內容落到可執行的 repo baseline。若沒有這一步，工程團隊在啟動時仍會面對結構空白、環境不一致、migration 無落點、skills 與 source manifests 無標準模板等問題，導致 Phase 1 交付速度下降與 scope drift 風險上升。

## 4. Success Criteria

本版本計劃交付成功，需同時滿足以下條件：

1. repo 結構完成，且和 engineering spec 一致
2. SQL migration baseline 存在且可審閱
3. Hermes-related scaffold 存在，並可作為後續技能撰寫起點
4. `.env.example`、source manifest、Langfuse compose 存在
5. operations runbook 與 README 能讓工程 owner 在 Day 1 啟動工作
6. 無任何明顯與既有 spec / overview / stakeholder brief 相衝突的內容

## 6. Architecture Alignment

本版本計劃嚴格遵循現有文件的 Phase 1 共識：

- Hermes 為 Phase 1 default runtime
- Firecrawl 為 Phase 1 default web backend
- Supabase Cloud 為 Phase 1 primary structured store
- Langfuse 為 Phase 1 observability layer
- GraphDB / BFF / frontend 均 defer 至 Phase 2 條件觸發後再考慮

## 9. Risks

| Risk | Impact | Mitigation |
|---|---|---|
| repo scaffold accidentally implies unsupported runtime behavior | future engineering confusion | keep artifacts scaffold-level where actual runtime details are unknown |
| fake Hermes implementation assumptions | invalid developer guidance | limit to config/skill/runbook scaffolds, avoid fabricating runtime code APIs |
| schema drift from engineering spec | execution confusion | derive all DB files directly from approved spec |
| premature Phase 2 leakage | scope creep | explicitly mark deferred items in README and runbook |

## 10. Design Pattern Intent

### Repository Pattern
- Signal: data persistence is a separate concern from runtime orchestration
- Alternative: keep only raw SQL files and no logical grouping
- Cost: slightly more structure in docs and file layout, but clearer ownership
- Decision: apply only as file organization guidance, not as application code implementation in Phase 1

### Adapter Pattern
- Signal: Firecrawl, Supabase, Langfuse are external systems with distinct contracts
- Alternative: document them separately with no abstraction language
- Cost: mild conceptual overhead
- Decision: only document integration boundaries, do not implement code adapters yet

### Current complexity does not justify additional design patterns or abstractions.
